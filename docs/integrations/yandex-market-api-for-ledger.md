# Яндекс Маркет Partner API — выжимка для детерминированного леджера

Корень документации: [API Яндекс Маркета для продавцов](https://yandex.ru/dev/market/partner-api/doc/ru/).

Этот файл — **сводка под кейс** «заказы → начисления → выплата пачкой на счёт → сверка с банком (Точка)» + **API‑уведомления**. Официальная дока первична; структура отчётов может меняться без предупреждения (прямо указано в справке).

---

## 1. Авторизация

- Токен **Api-Key** в заголовке: `Api-Key: <token>`.  
- Создание и права: [Создание и использование API-Key-токена](https://yandex.ru/dev/market/partner-api/doc/ru/concepts/api-key).  
- Для финансовых отчётов в кабинете токену нужны соответствующие **доступы к группам методов** (в доке — `finance-and-accounting` и т.д.; точный список — в разделе доступов при создании токена).

Базовый хост API (из методов отчётов и заказов): `https://api.partner.market.yandex.ru`

## 2. Заказы (операционный слой)

### 2.1 Список / детали заказов

- **POST** `v1/businesses/{businessId}/orders` — информация о заказах кабинета, фильтры по датам (макс. диапазон **30 дней**), пагинация `page_token` / `limit`.  
  Дока: [Information about orders](https://yandex.ru/dev/market/partner-api/doc/en/reference/orders/getBusinessOrders) (EN; RU обычно зеркалится в том же дереве `reference/orders`).

Связь с леджером: события **`MP_ORDER`**, статусы, `orderId`, `campaignId`, `externalOrderId` (если прокидываете свой id — см. метод передачи внешнего id в доке по уведомлениям).

### 2.2 API‑уведомления (рекомендовано)

- Маркет шлёт **POST** на ваш URL `POST /notification` (HTTPS).  
- Типы, важные для леджера: `ORDER_CREATED`, `ORDER_UPDATED`, `ORDER_STATUS_UPDATED`, `ORDER_CANCELLED`, `ORDER_RETURN_CREATED`, …  
- После уведомления — подтянуть полные данные: **POST** `v1/businesses/{businessId}/orders` по `orderIds`.  
- Таймаут ответа: **10 с** (для `PING` — **1 с**). При ошибках — повторные отправки; при длительной недоступности интеграция может отключиться (см. доку).

Дока: [Получение уведомлений](https://yandex.ru/dev/market/partner-api/doc/ru/push-notifications/reference/sendNotification), обзор: [step-by-step orders](https://yandex.ru/dev/market/partner-api/doc/en/step-by-step/orders-receive).

## 3. Финансы: отчёт по платежам (сшивка с банком)

Для **выплат на расчётный счёт**, удержаний и привязки к **платёжному поручению** используется **отчёт по платежам** (united netting).

- **POST** `v2/reports/united-netting/generate` — запуск генерации.  
- Типы (взаимоисключающие поля в одном запросе):  
  - **«О платежах за период»** — `dateFrom` и `dateTo`; в отчёт попадают платежи, **выплаченные и начисленные** в период (важно: один перевод может попасть в **два** месяца по правилам Маркета — см. доку).  
  - **«О платежном поручении»** — `bankOrderId` и `bankOrderDateTime` (точечная сверка с **одной** операцией в выписке Точки).  
- **GET** `v2/reports/info/{reportId}` — статус и **ссылка на файл**; ссылка живёт **60 минут** — скачать и сохранить у себя.

Дока: [Отчет по платежам](https://yandex.ru/dev/market/partner-api/doc/ru/reference/reports/generateUnitedNettingReport).

Колонки отчёта (имена в CSV/JSON): в доке перечислены, среди прочего, **`TRANSACTION_ID`**, **`TRANSACTION_SUM`**, **`BANK_ORDER_DATE`**, сумма/номер платёжного поручения, **`PAYMENT_STATUS`**, комментарии и др. — использовать как **канонические поля** при парсинге в `MP_PAYOUT` / строки к выплате.

Лимиты генерации: в доке указаны лимиты запросов (зависят от подписки; есть старые/новые даты — смотреть актуальную страницу).

## 4. Операционный отчёт по заказам (маржа, платежи покупателя)

- **POST** генерация «Отчёт по заказам» — в доке: [Отчет по заказам](https://yandex.ru/dev/market/partner-api/doc/ru/reference/reports/generateUnitedOrdersReport).

Полезные для леджера группы колонок (из описания листов):

- идентификаторы заказа, даты, суммы;  
- **`buyerPaymentAmount`**, **`buyerPaymentBankOrderId`**, **`buyerPaymentBankOrderDate`** и аналоги по скидкам Маркета / СберСпасибо / Плюс;  
- **`incomeWithoutServices`**, комиссии, **`buyerPayment`**, строки услуг.

Это помогает связать **заказ** с **денежным потоком** и с **выплатой**, не дублируя УСН‑базу без правила в `rules_version`.

## 6. Маппинг на события `provable-ops` (наш кейс)

| Источник Маркета | Событие леджера | Комментарий |
|------------------|-----------------|-------------|
| Уведомление + заказ | `MP_ORDER` | Минимум из push; детали — из POST orders |
| Отчёт / API (если есть начисление на баланс) | `MP_BALANCE_CREDIT` | Если в данных явно выделяется |
| Строка «отчёт по платежам» / выплата | `MP_PAYOUT` | Агрегат за день; для банка — сумма многих заказов |
| `bankOrderId` + дата из отчёта | `MATCH` → `BANK_IN` | Сверка с выпиской Точки по `paymentId` / назначению |

Внутренние шаги (ваучер, поставщик, email) остаются **`source = internal`** / **`supplier`** — см. `cases/usn-marketplace-digital/SPEC.md`.
