# Кейс: УСН «доходы» + маркетплейс + цифровые товары (поставщик)

Первый **детерминированный леджер** строится на твоём личном сценарии.

**Флоу:** продажа на маркетплейсе (Яндекс) → ваучер у вашего бэкенда → передача на маркетплейс → доставка клиенту → начисление на **баланс** продавца на маркетплейсе → обмен ваучера у вас → покупка у поставщика по API → гифт‑карта на email клиента → **ежедневная пачка** от Яндекса на расчётный счёт (одна операция в банке на много заказов; разрез — из API маркетплейса).

**Уже ведётся бизнес:** продажи на Маркете, заказы у поставщика (EZ PIN) и переводы в Точку могли быть **до** первого запуска журнала — тогда нужен **backfill** за выбранное окно дат и та же логика сверки, что и «вперёд» (см. [SPEC.md §0b](./SPEC.md)).

**Прокси и SKU:** запросы к EZ PIN идут через ваш прокси с маппингом **маркет ↔ внутренний ↔ EZ PIN**; версия таблицы (**`sku_map_version`**) и три поля SKU входят в канонический `payload` для Merkle (см. [SPEC.md §2.3b](./SPEC.md), [FIELD_MAP.md](./FIELD_MAP.md)).

Источники: **API маркетплейса**, **банк** (API/выписка), **API поставщика**, плюс события **`internal`** с вашего бэкенда.

Спецификация: [SPEC.md](./SPEC.md).  
Примеры событий: [examples/sample-events.jsonl](./examples/sample-events.jsonl).

**Важно:** налоговая база УСН и момент дохода — вопрос **версии правил** (`rules_version`) и консультанта; леджер гарантирует **воспроизводимость** и **сверку потоков**, а не «одобрение ФНС».
