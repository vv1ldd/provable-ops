# SPEC: детерминированный журнал (УСН доходы · маркетплейс · цифровые товары)

**Версия:** `rules_version = usn-mp-digital-0.1.0`  
**Статус:** черновик под первую реализацию.

---

## 1. Допущения (заполни под себя)

| Поле | Значение (пример) |
|------|-------------------|
| Субъект | ИП или ООО (влияет на взносы/отчётность — в v0.1 только пометка `entity_kind`) |
| Режим | УСН «доходы» |
| Деятельность | Продажа через маркетплейс, товар **цифровой**, закупка у поставщика по API |
| Валюта учёта | RUB (если мультивалюта — отдельное расширение) |

**Юридическое:** для УСН «доходы» налоговая база обычно строится из **доходов**, признаваемых в соответствии с НК; **себестоимость поставщика** критична для **маржи и бухгалтерской прибыли**, но **не уменьшает** ставку УСН «доходы» как правило. В журнале она всё равно нужна, чтобы **сшить** деньги, обязательства перед поставщиком и отчёты маркетплейса.

**Маркетплейс:** комиссии, удержания, выплаты — в `rules_version` фиксируется, что считается **выплатой тебе** (сумма к зачислению) и как это мэчится с банком.

## 2. Источники (три контура)

### 2.1 Маркетплейс (API)

Нормализуем в **события** (имена условные — подставь поля реального API):

- продажа / заказ (идентификатор заказа, дата, сумма к оплате покупателем, статус);  
- возврат;  
- **выплата продавцу** (payout id, период агрегата, сумма к перечислению, комиссия маркетплейса);  
- корректировки / споры (если есть).

**Идемпотентность:** каждое сырьё API → `external_ref` + `source = marketplace` + `fetched_at`.

### 2.2 Банк (API или выгрузка)

- входящие зачисления (ожидаемые от маркетплейса);  
- исходящие (оплата поставщику, налоги, взносы — по мере включения в кейс).

**Идемпотентность:** `bank_tx_id` + `value_date` + `amount` + `currency` + нормализованное `purpose`.

### 2.3 Поставщик цифровых товаров (API)

- выдача/активация лицензии (product id, quantity, unit_price, заказ);  
- счёт / акт в электронном виде (если API отдаёт);  
- статус оплаты с их стороны (если есть).

**Идемпотентность:** `supplier_event_id` + `source = supplier`.

### 2.4 Опционально: 1С

Если позже подключаешь выгрузку из 1С — те же события могут дублироваться как `source = accounting` для **тройной** сверки. В v0.1 можно **не** тянуть 1С, а считать журнал «истиной витрины» до появления бухгалтерского контура.

## 3. Канонизация (детерминизм)

1. Все суммы → `decimal` строкой с фиксированным масштабом (например 2 знака после запятой для RUB).  
2. Все моменты времени → UTC ISO8601 `...Z` + при необходимости `local_date` для налогового периода.  
3. Сортировка событий внутри периода Merkle:  
   `(source ASC, external_ref ASC)` — зафиксировать в коде и не менять без bump `rules_version`.  
4. `payload_canonical` = стабильный JSON без пробельных вариаций (или CBOR — выбрать одно на v0.1).

---

## 4. Типы событий (MVP)

| `event_type` | Смысл |
|--------------|--------|
| `MP_ORDER` | Заказ/продажа на маркетплейсе (до выплаты) |
| `MP_PAYOUT` | Выплата маркетплейса (агрегат или строка) |
| `BANK_IN` | Входящее на расчётный счёт |
| `BANK_OUT` | Исходящее (оплата поставщику и т.д.) |
| `SUP_DELIVERY` | Факт выдачи цифрового товара от поставщика |
| `MATCH` | Результат автоматической сверки (связка сущностей) |
| `USN_PERIOD_ROLLUP` | Агрегат доходов за квартал по **выбранным** в правилах событиям |

Поле каждого события (минимум):

- `event_id` (UUID v4 или ULID — один формат на весь журнал)  
- `event_type`  
- `occurred_at` (момент события в бизнес‑смысле)  
- `ingested_at` (момент попадания в систему)  
- `source`  
- `external_ref`  
- `payload` (объект → потом канонизируется в байты)

## 5. Сверка (логика v0.1)

### 5.1 `MP_PAYOUT` ↔ `BANK_IN`

- Сумма в пределах допуска `epsilon_money` (например 0.01 RUB) **или** явное правило «маркетплейс удержал комиссию» → ожидаемая сумма к зачислению = `mp_payout.net` vs `bank_in.amount`.  
- Окно дат: `payout_date` и `bank value_date` в пределах `window_days` (например 0–5).  
- Назначение платежа / референс: если банк передаёт код выплаты — жёсткий мэтч.

Результат → событие `MATCH` с полями: `left_event_id`, `right_event_id`, `match_kind`, `confidence`, `notes`.

### 5.2 `SUP_DELIVERY` ↔ `BANK_OUT` (или счёт к оплате)

- По сумме и контрагенту / назначению; окно дат.  
- Если оплата **предоплата** — цепочка из нескольких `BANK_OUT` к одному поставщику (версия правил).

### 5.3 `MP_ORDER` ↔ `SUP_DELIVERY`

- Связка по **внутреннему sku / offer id**, если маркетплейс и поставщик согласованы по артикулу; иначе ручной mapping‑таблица `marketplace_sku → supplier_product_id` (версионируется).

## 6. УСН «доходы» (rollup в журнале)

На `rules_version = usn-mp-digital-0.1.0`:

- В квартальный `USN_PERIOD_ROLLUP` попадают только события, помеченные как **«доход для УСН»** согласно таблице в правилах (например: **зачисление от маркетплейса** после `MATCH` с `MP_PAYOUT`; или **выручка по заказам** — **не оба**, выбрать одно и зафиксировать).  
- Ставка УСН (например 6%) хранится как константа в правилах на период (ставка может меняться — только через новый `rules_version`).

**Обязательная оговорка:** момент признания дохода при работе через маркетплейс (агент / не агент, удержания) — **вынести в чеклист с налоговым консультантом** и одной строкой в README кейса.

## 7. Merkle и commitment за период

- Период: **календарный день** или **квартал** для MVP (день — для отладки, квартал — для налогового смысла).  
- Листья: `leaf_hash = SHA256( event_id || rules_version || payload_canonical_bytes )` (алгоритм зафиксировать в `Merkle Profile v1` репозитория).  
- Публикуется `period_root` + `rules_version` + количество событий.

## 8. Следующие шаги разработки

1. Выписать **реальные поля** твоего API маркетплейса (без секретов) → таблица маппинга в `FIELD_MAP.md`.  
2. То же для банка (формат выписки / JSON ответа).  
3. То же для поставщика.  
4. Репозиторий кода: ingestion workers + SQLite/Postgres append-only + CLI `ledger verify-period`.  

Когда будут три сэмпла (по одному JSON на источник), можно добавить **золотой тест**: вход → ожидаемый `period_root`.
