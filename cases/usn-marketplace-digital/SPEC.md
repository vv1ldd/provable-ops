# SPEC: детерминированный журнал (УСН доходы · маркетплейс · цифровые товары)

**Версия:** `rules_version = usn-mp-digital-0.2.1`  
**Статус:** черновик под первую реализацию (обновлено под фактический флоу: ваучер → маркетплейс → баланс → обмен → поставщик → email; выплата пачкой на расчётный счёт). В **0.2.1** добавлена явная модель **ретроспективы** (операции уже были до запуска журнала).

---

## 0. Реальный бизнес‑флоу (как сейчас)

Цепочка **на стороне маркетплейса (Яндекс)**:

1. Продажа на маркетплейсе.  
2. **Генерация ваучера** вашим бэкендом.  
3. Передача ваучера на маркетплейс.  
4. Доставка маркетплейсом до клиента.  
5. **Начисление суммы покупки на баланс** продавца на маркетплейсе (внутренний баланс, не банк).

Цепочка **на вашем сервисе и у поставщика**:

6. **Обмен ваучера** на вашем сервисе.  
7. Бэкенд запрашивает у **API поставщика** код / товар.  
8. **Покупка у поставщика по API** (транзакция).  
9. Отправка **гифт‑карты** вашим бэкендом на email клиента.

**Банк:** маркетплейс перечисляет на **расчётный счёт пачкой** (например **один платёж в день**). Сумма в банке — **агрегат**; разрез по заказам берётся из **API маркетплейса** (отчёты по заказам / выплатам / балансу — конкретные эндпоинты зафиксировать в `FIELD_MAP.md`).

**Следствие для леджера:** одна строка `BANK_IN` за день должна **сходиться** с суммой множества нормализованных событий маркетплейса за тот же **расчётный период выплаты** (см. §5.4). Внутренние шаги 2–9 дают **полную трассировку** до поставщика и клиента.

## 0b. Ретроспектива: продажи, EZ PIN и перевод в банк уже были

Типичный запуск не с «первого пустого дня», а когда **уже существуют**:

- продажи и начисления на маркетплейсе (история заказов / отчёты);  
- покупки у поставщика по API (история заказов `reference_code` / период в EZ PIN);  
- **как минимум одна** выплата маркетплейса пачкой на расчётный счёт в **Точке** (`BANK_IN`).

**Инвариант:** журнал строится **задним числом** теми же правилами канонизации и сверки (§3–§5); меняется только **окно загрузки** и порядок работ.

### Порядок backfill (практический)

1. **Зафиксировать окно** `T0..T1` (например первый месяц с продажами или целый квартал). Один и тот же интервал по возможности для всех трёх контуров, с поправкой на то, что дата **зачисления в банке** может быть **T+1** относительно даты выплаты в отчёте Маркета — использовать `window_days` из §5.1 / §5.4.  
2. **Маркетплейс:** выгрузить заказы за период (POST orders с датами; при необходимости — **отчёт по платежам** united-netting за `T0..T1` или точечно по `bankOrderId` + `bankOrderDateTime` для строки, совпадающей с выпиской Точки).  
3. **Банк:** Init/Get Statement за период, **покрывающий** фактические `value_date` входящих от Маркета (часто чуть шире `T0..T1` по календарю).  
4. **Поставщик:** список заказов за период (`GET /orders/` с датами, если API поддерживает) + детали по нужным `reference_code`.  
5. **Internal:** события ваучера / email за прошлое — только если есть **логи**; иначе в журнале допускается `gap` в трассировке с пометкой в `notes` (не выдумывать `occurred_at`).

**Сверка «уже был перевод Яндекс → Точка»:** сначала найти строку `BANK_IN` в выписке (сумма, назначение, дата), затем **привязать** к ней агрегат `MP_PAYOUT` / строки united-netting (см. §5.4), затем разложить по заказам. Это тот же `MATCH`, что и для будущих дней, только `ingested_at` позже бизнес‑даты.

## 1. Допущения (заполни под себя)

| Поле | Значение (пример) |
|------|-------------------|
| Субъект | ИП или ООО (влияет на взносы/отчётность — в v0.1 только пометка `entity_kind`) |
| Режим | УСН «доходы» |
| Деятельность | Продажа через маркетплейс, товар **цифровой**, закупка у поставщика по API |
| Валюта учёта | RUB (если мультивалюта — отдельное расширение) |

**Юридическое:** для УСН «доходы» налоговая база обычно строится из **доходов**, признаваемых в соответствии с НК; **себестоимость поставщика** критична для **маржи и бухгалтерской прибыли**, но **не уменьшает** ставку УСН «доходы» как правило. В журнале она всё равно нужна, чтобы **сшить** деньги, обязательства перед поставщиком и отчёты маркетплейса.

**Маркетплейс:** комиссии, удержания, выплаты — в `rules_version` фиксируется, что считается **выплатой тебе** (сумма к зачислению) и как это мэчится с банком.

## 2. Источники (три контура)

### 2.1 Маркетплейс (API Яндекса)

Нормализуем в **события** (имена полей — из реального API, см. `FIELD_MAP.md`):

- заказ / продажа;  
- статусы доставки / выдачи покупателю;  
- **начисление на баланс продавца** (если API отдаёт отдельно от банковской выплаты);  
- **выплата с баланса на расчётный счёт** (агрегат за день/период) + **детализация по заказам** за тот же период (чтобы сшить с пачкой в банке);  
- возвраты, споры, корректировки.

**Идемпотентность:** каждое сырьё API → `external_ref` + `source = marketplace` + `fetched_at`.

### 2.1b Ваш бэкенд (`source = internal`)

События, которых нет у маркетплейса/банка как отдельного API, но они **критичны для трассировки**:

- выпуск ваучера;  
- передача ваучера на маркетплейс;  
- факт обмена ваучера на сервисе;  
- вызов API поставщика и подтверждение покупки;  
- отправка гифт‑карты на email (лог: message id / шаблон времени, **без** содержимого письма и PII в публичных хэшах — политика версионируется).

`external_ref` здесь: например `int:voucher:{uuid}`, `int:redemption:{id}`.

### 2.2 Банк (API или выгрузка)

- входящие зачисления (ожидаемые от маркетплейса);  
- исходящие (оплата поставщику, налоги, взносы — по мере включения в кейс).

**Идемпотентность:** `bank_tx_id` + `value_date` + `amount` + `currency` + нормализованное `purpose`.

### 2.3 Поставщик цифровых товаров (API)

- выдача/активация лицензии (product id, quantity, unit_price, заказ);  
- счёт / акт в электронном виде (если API отдаёт);  
- статус оплаты с их стороны (если есть).

**Идемпотентность:** `supplier_event_id` + `source = supplier`.

### 2.4 Опционально: 1С

Если позже подключаешь выгрузку из 1С — те же события могут дублироваться как `source = accounting` для **тройной** сверки. В v0.1 можно **не** тянуть 1С, а считать журнал «истиной витрины» до появления бухгалтерского контура.

## 3. Канонизация (детерминизм)

1. Все суммы → `decimal` строкой с фиксированным масштабом (например 2 знака после запятой для RUB).  
2. Все моменты времени → UTC ISO8601 `...Z` + при необходимости `local_date` для налогового периода.  
3. Сортировка событий внутри периода Merkle:  
   `(source ASC, external_ref ASC)` — зафиксировать в коде и не менять без bump `rules_version`.  
4. `payload_canonical` = стабильный JSON без пробельных вариаций (или CBOR — выбрать одно на v0.1).

## 4. Типы событий (MVP)

| `event_type` | Смысл |
|--------------|--------|
| `MP_ORDER` | Продажа / заказ на маркетплейсе |
| `MP_BALANCE_CREDIT` | Начисление на баланс продавца на маркетплейсе (если выделяется в данных) |
| `MP_PAYOUT` | Выплата маркетплейса на расчётный счёт (часто **агрегат за день**) |
| `INT_VOUCHER_ISSUED` | Ваучер сгенерирован вашим бэкендом |
| `INT_VOUCHER_TO_MP` | Ваучер передан на маркетплейс |
| `INT_VOUCHER_REDEEMED` | Обмен ваучера на вашем сервисе |
| `SUP_PURCHASE` | Покупка/выдача у поставщика по API (код, гифт‑карта) |
| `INT_GIFT_EMAIL_SENT` | Гифт‑карта отправлена на email клиента (метаданные без PII в публичном слое) |
| `BANK_IN` | Входящее на расчётный счёт (в т.ч. **одна строка на день** от маркетплейса) |
| `BANK_OUT` | Исходящее (оплата поставщику и т.д.) |
| `MATCH` | Результат сверки (связка сущностей) |
| `USN_PERIOD_ROLLUP` | Агрегат доходов за квартал по **выбранным** в правилах событиям |

Поле каждого события (минимум):

- `event_id` (UUID v4 или ULID — один формат на весь журнал)  
- `event_type`  
- `occurred_at` (момент события в бизнес‑смысле)  
- `ingested_at` (момент попадания в систему)  
- `source`  
- `external_ref`  
- `payload` (объект → потом канонизируется в байты)

## 5. Сверка (логика v0.1)

### 5.1 `MP_PAYOUT` ↔ `BANK_IN`

- Сумма в пределах допуска `epsilon_money` (например 0.01 RUB) **или** явное правило «маркетплейс удержал комиссию» → ожидаемая сумма к зачислению = `mp_payout.net` vs `bank_in.amount`.  
- Окно дат: `payout_date` и `bank value_date` в пределах `window_days` (например 0–5).  
- Назначение платежа / референс: если банк передаёт код выплаты — жёсткий мэтч.

Результат → событие `MATCH` с полями: `left_event_id`, `right_event_id`, `match_kind`, `confidence`, `notes`.

### 5.2 `SUP_DELIVERY` ↔ `BANK_OUT` (или счёт к оплате)

- По сумме и контрагенту / назначению; окно дат.  
- Если оплата **предоплата** — цепочка из нескольких `BANK_OUT` к одному поставщику (версия правил).

### 5.3 `MP_ORDER` ↔ `INT_VOUCHER_*` ↔ `SUP_PURCHASE`

- `order_id` / `offer_id` ↔ `voucher_id` (ваш UUID) — в вашей БД;  
- `INT_VOUCHER_REDEEMED` ↔ `SUP_PURCHASE` по `voucher_id` или `redemption_id`;  
- при необходимости mapping `marketplace_sku → supplier_product_id` (версионируется).

### 5.4 Пачка на расчётный счёт: `MP_PAYOUT` (сумма заказов) ↔ одна `BANK_IN`

- За календарный день \(D\) маркетплейс отдаёт **список заказов**, попавших в выплату, и **итог к перечислению** `S_mp(D)`.  
- Банк даёт **одну** (или несколько) операций `BANK_IN` с датой \(D'\) (часто \(D' = D\) или \(D+1\)).  
- Правило: \(\sum \text{net по заказам в периоде выплаты} \approx \text{amount}(BANK\_IN)\) с допуском `epsilon_money` и учётом **удержаний** из API.  
- Если API отдаёт **идентификатор реестра выплаты** — мэтчить `BANK_IN.purpose` с этим id.  
- Результат: один `MATCH` типа `DAILY_BATCH_BANK_IN` с ссылкой на **множество** `MP_ORDER` / `MP_PAYOUT` line items (хранить как массив id или отдельные `MATCH` строки — зафиксировать в `rules_version`).

## 6. УСН «доходы» (rollup в журнале)

На `rules_version = usn-mp-digital-0.2.1`:

- В квартальный `USN_PERIOD_ROLLUP` попадают только события, помеченные как **«доход для УСН»** согласно таблице в правилах (например: **зачисление от маркетплейса** после `MATCH` с `MP_PAYOUT`; или **выручка по заказам** — **не оба**, выбрать одно и зафиксировать).  
- Ставка УСН (например 6%) хранится как константа в правилах на период (ставка может меняться — только через новый `rules_version`).

**Обязательная оговорка:** момент признания дохода при работе через маркетплейс (агент / не агент, удержания) — **вынести в чеклист с налоговым консультантом** и одной строкой в README кейса.

## 7. Merkle и commitment за период

- Период: **календарный день** или **квартал** для MVP (день — для отладки, квартал — для налогового смысла).  
- Листья: `leaf_hash = SHA256( event_id || rules_version || payload_canonical_bytes )` (алгоритм зафиксировать в `Merkle Profile v1` репозитория).  
- Публикуется `period_root` + `rules_version` + количество событий.
